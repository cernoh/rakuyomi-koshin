    **`backend/server/src/track/qr.rs`** — QR code generation:

    ```rust
    use crate::AppError;

    /// Encode a URL as a 300×300px greyscale PNG QR code.
    /// Returns the PNG bytes.
    pub fn encode_url_to_qr_png(url: &str) -> Result<Vec<u8>, AppError> {
        let code = qrcode::QrCode::new(url.as_bytes())
            .map_err(|e| AppError::Other(e.into()))?;
        let image = code.render::<qrcode::render::image::Luma<u8>>()
            .min_dimensions(300, 300)
            .build();
        let mut png_bytes = Vec::new();
        image.write_to(&mut std::io::Cursor::new(&mut png_bytes), image::ImageFormat::Png)
            .map_err(|e| AppError::Other(e.into()))?;
        Ok(png_bytes)
    }
    ```

    **Important**: The `image` crate for `ImageFormat::Png` and `Luma<u8>` type. If `image` isn't a direct dep of server (it's a shared crate dep), the `qrcode` crate with `image` feature re-exports what's needed. The `render::image::Luma<u8>` path uses `qrcode`'s feature-gated image module. If `write_to` requires `image` crate directly, use `image` types from the shared crate path — they're the same `image 0.25` crate. The `image` types (`image::ImageFormat`, `image::Luma`) are available if `image` is a dependency. Since server doesn't list `image` as dep but qrcode with image feature pulls it in, the types may need explicit path. Use `::image::` path if needed, or use the types through qrcode's re-exports.

    If compilation issues arise with `image` types, the simplest approach: render the QR to a `Vec<bool>` matrix and manually construct a PNG via the `image` crate brought in by the `qrcode/image` feature. The `image` crate should be available as a transitive dep.

    **`backend/server/src/track/routes.rs`** — HTTP handlers following the established manga/routes.rs pattern:

    ```rust
    use std::sync::Arc;
    use axum::{
        Router,
        extract::{Path, State as StateExtractor},
        routing::{get, post, delete},
        Json,
    };
    use serde::{Deserialize, Serialize};
    use tokio::sync::Mutex;
    use shared::track::types::{AuthToken, TrackerService};
    use crate::{
        state::{State, TrackState},
        track::auth::{AniListAuth, MalAuth, TrackerAuth},
        AppError,
    };
    use super::qr::encode_url_to_qr_png;

    pub fn routes() -> Router<State> {
        Router::new()
            .route("/services", get(list_services))
            .route("/{tracker}/auth-url", post(generate_auth_url))
            .route("/qr/{qr_id}", get(get_qr_code))
            .route("/{tracker}/auth", post(submit_auth).delete(clear_auth))
            .route("/{tracker}/status", get(check_status))
    }
    ```

    **Route handler signatures and logic** (all return `Result<Json<...>, AppError>`):

    1. **GET /track/services** — `list_services`
       - Extracts `StateExtractor(State { database, .. })`
       - Queries `tracker_auth` table for all rows (select tracker_id)
       - Returns JSON `Vec<ServiceStatus>` where `ServiceStatus { tracker_id: String, logged_in: bool }`
       - Known tracker IDs: `["anilist", "myanimelist"]`
       - For each known tracker, check if a row exists in tracker_auth
       - Use `sqlx::query_as::<_, (String,)>(r#"SELECT tracker_id FROM tracker_auth"#).fetch_all(...)` or similar
       - Response DTO: `#[derive(Serialize)] struct ServiceStatus { tracker: String, logged_in: bool }`

    2. **POST /track/{tracker}/auth-url** — `generate_auth_url`
       - Extracts `Path(tracker): Path<String>` and `StateExtractor(State { track_state, .. })`
       - Validates tracker is `"anilist"` or `"myanimelist"`, return 400 if not
       - For `"anilist"`: call `AniListAuth::new().generate_auth_url().await` → `(url, _state)`
       - For `"myanimelist"`:
         - Call `MalAuth::new().generate_auth_url().await` → `(url, state)`
         - Store `PkceSession { code_verifier, tracker_id: "myanimelist", created_at }` in `track_state` keyed by `state`
       - Generate QR PNG from the URL
       - Return `Json(serde_json::json!({ "url": url_string, "qr_id": state_or_empty }))`
       - Where `qr_id` is the state for MAL (used to retrieve QR from /track/qr/{qr_id}) or empty for AniList

    3. **GET /track/qr/{qr_id}** — `get_qr_code`
       - Extracts `Path(qr_id): Path<String>` and `StateExtractor(State { track_state, .. })`
       - Looks up the PkceSession in track_state by qr_id (state string)
       - If found, generates QR from the stored auth URL — BUT the URL isn't stored in PkceSession. Re-design: either store the URL in PkceSession, or regenerate it.
       - Better approach: PkceSession should also store the `auth_url: String` field so it can be retrieved for QR generation. Update PkceSession struct in Plan 1's state.rs to add `pub auth_url: String`. This is a minor change; alternatively, generate QR from the auth URL that was already generated in step 2 and store it alongside the session.
       - **CRITICAL UPDATE**: Modify `PkceSession` in `backend/server/src/track/state.rs` to add `pub auth_url: String` field. This allows QR regeneration by session lookup.
       - If no session found (expired or invalid), return `AppError::NotFound`
       - Generate QR PNG via `encode_url_to_qr_png(&session.auth_url)`
       - Return with content type `image/png`. Use axum's response type:
         ```rust
         let png = encode_url_to_qr_png(&session.auth_url)?;
         Ok((
             [("Content-Type", "image/png")],
             png,
         ))
         ```
       - Return type: `Result<([(&str, &str); 1], Vec<u8>), AppError>` — this works because `Vec<u8>` implements `IntoResponse`

    4. **POST /track/{tracker}/auth** — `submit_auth`
       - Accepts JSON body. For AniList (implicit): `{ "token": "..." }`. For MAL (PKCE): `{ "code": "...", "state": "..." }`
       - Request body DTO: `#[derive(Deserialize)] struct SubmitAuthBody { token: Option<String>, code: Option<String>, state: Option<String> }`
       - Validate that at least token or code+state are present
       - For `"anilist"`:
         - Call `AniListAuth::new().verify_token(&AuthToken { access_token: token, .. }).await`
         - If verification succeeds, store token_json in `tracker_auth` table via sqlx
         - Use `INSERT OR REPLACE INTO tracker_auth (tracker_id, token_json, created_at) VALUES (?, ?, datetime('now'))`
       - For `"myanimelist"`:
         - Look up PkceSession by state from track_state (get_and_remove)
         - Call `MalAuth::new().exchange_code(code, &session.code_verifier).await`
         - Store the resulting AuthToken in tracker_auth table (token_json serialized via serde_json)
         - If token has expires_in, set expires_at = datetime('now', '+{seconds} seconds')
       - Return `Json(serde_json::json!({ "success": true }))`

    5. **DELETE /track/{tracker}/auth** — `clear_auth`
       - Extracts `Path(tracker): Path<String>` and `StateExtractor(State { database, .. })`
       - Validates tracker is known
       - `DELETE FROM tracker_auth WHERE tracker_id = ?`
       - Return `Json(serde_json::json!({ "success": true }))`

    6. **GET /track/{tracker}/status** — `check_status`
       - Extracts `Path(tracker): Path<String>` and `StateExtractor(State { database, .. })`
       - Query `SELECT token_json FROM tracker_auth WHERE tracker_id = ?`
       - If no row → `{ "logged_in": false, "username": null }`
       - If row exists, try to verify (optional — can just return logged_in without verification for speed)
       - Return `Json(serde_json::json!({ "logged_in": true, "username": ... }))` or minimal `{ "logged_in": true }`

    **SQL queries pattern**: Use `sqlx::query()` with `database.pool()` (or `database.get_pool()`) — follow the existing pattern from the shared crate. In route handlers, access the pool via `&StateExtractor(State { database, .. })`. The `Database` type provides access methods. For Phase 1, the simplest pattern is direct sqlx queries on the database pool.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → /track/* endpoints | All endpoints served on localhost/UDS only; no external network exposure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07 | Information Disclosure | GET /track/qr/{qr_id} | accept | QR ID = PKCE state (random 64-char hex). Brute-force infeasible. 15-min TTL. Local-only server. |
| T-01-08 | Tampering | POST /track/{tracker}/auth | mitigate | Validate tracker param is known service. Validate body has required fields. Token stored only after API verification. |
| T-01-09 | Repudiation | DELETE /track/{tracker}/auth | accept | Credential deletion is intentional user action. No audit log needed for local service. |
| T-01-10 | Spoofing | PKCE state reuse | mitigate | `get_and_remove()` atomically consumes state — one-time use only. Replay of same state fails. |
| T-01-SC | Tampering | qrcode crate | mitigate | Already verified OK in Plan 2's Package Legitimacy Audit. |
</threat_model>
